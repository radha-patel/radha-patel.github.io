What is the MIT License?

Saturday March 2nd, 2024

If you’ve been involved in open-source development, you’ve likely seen code or documentation files marked with the “MIT License.” For a good amount of time, I didn't quite know what this meant or what the purpose was. Why was my university licensing nearly all the open-source code I ever looked at???

The MIT License was first developed at MIT (crazy! you would have never guessed it!) in the late 1980s as open source code was becoming more prevalent. The purpose of the MIT License is to grant permission to users to reuse code for any purpose, even if the code is originally part of proprietary software—that is, as long as users include the original copy of the MIT license in their distribution. The MIT License also allows for commercial use. With it, however, the software and license owner cannot be charged for damages. To use the MIT license, all developers need to do is copy the license text and include it in source files.

According to this source, in 2019 researchers found that 27% of all open-source licenses on Github were using the MIT License, making it the most popular open source license that exists. There are many other licenses, most of which can be categorized in two groups: copyleft or permissive.

In a copyleft license, code derived from the original open source code inherits its license terms. The most popular copyleft open source licenses include GPL, AGPL, EPL, and Mozilla, as described below and in more detail in this article by Synk.

  • If you use GNU General Public License (GPL) code in your software (e.g. by using a GPL library), and distribute your application, all your source code must be made available under the same GPL license. A loophole exists, however, for software that is made available over the network only (i.e. not explicitly “distributed”).
  • Affero GPL (AGPL) closes the loophole in GPL by also including a remote interaction clause that triggers the GPL license for any software used over a network.
  • Eclipse Public License (EPL) enables software developed using EPL, non-EPL, and proprietary code to be combined and sub-licensed as long as non-EPL elements reside independently as separate modules.
  • The Mozilla Public License (MPL) makes it easy to modify and use MPL-licensed code in closed-source and/or proprietary software, as long as any code licensed under the MPL is kept in separate files and these files are distributed with the software.

On the other hand, a permissive license is more lax and provides more freedom for reuse, modification, and distribution. Unlike for the copyleft license, software with a permissive license does not have to disclose its source, have the same license as its sources, or typically state changes. The most popular permissive licenses are Apache, MIT, BSD, and Unilicense.

  • The Apache License enables derivative works, larger projects, or modifications to carry different licensing terms when distributed. It also contains a patent grant.
  • The MIT License, as I’ve described above, allows anyone to do whatever they wish with the original code, as long as the original copyright and license notice is included.
  • The Berkeley Source Distribution (BSD) License allows larger or licensed works to be distributed without source code and under different license terms
  • The Unlicense is equivalent to making open source open to public domain. There are no conditions.

Evidently, there are a lot of different licenses out there and it’s likely more will get created to handle the use of AI in writing code. For now, however, using the MIT License seems to be a safe bet!
